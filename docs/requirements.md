# 다이어트/식단기록(dietlog) 요구사항 정의서

- 작성자: 정동욱
- 작성일: 2026-07-23
- 프로젝트 기간: 5일

## 1. 프로젝트 개요
- 한 줄 소개: 개인이 끼니별 섭취 음식과 칼로리를 기록하고, 끼니별·음식군별·월별 통계를 보는 식단기록 웹 서비스
- 대상 사용자: 로그인한 개인 사용자(내 데이터는 나만 접근)
- 목표: 부트캠프 전 과정 복습 + EC2 실배포까지 완주

## 2. 유저 스토리
| # | 유저 스토리 (~로서 나는 ~하고 싶다, 왜냐하면 ~) | 연결되는 기능 |
|---|-----------------------------------------------|---|
| US-1 | 사용자로서 나는 이메일과 비밀번호로 회원가입/로그인하고 싶다, 왜냐하면 내 식단 기록을 나만 보고 싶기 때문이다. | 기능 1 (회원가입/로그인) |
| US-2 | 사용자로서 나는 먹은 음식과 칼로리를 등록하고 싶다, 왜냐하면 하루에 얼마나 섭취했는지 기록해야 하기 때문이다. | 기능 3 (식단 기록 CRUD) |
| US-3 | 사용자로서 나는 등록한 식단 기록을 수정·삭제하고 싶다, 왜냐하면 칼로리나 음식명을 잘못 입력할 수 있기 때문이다. | 기능 3 (식단 기록 CRUD) |
| US-4 | 사용자로서 나는 기록을 끼니(아침/점심/저녁/간식)와 음식군(한식/양식/분식 등)으로 분류하고 싶다, 왜냐하면 언제 무엇을 많이 먹는지 알고 싶기 때문이다. | 기능 2 (카테고리) |
| US-5 | 사용자로서 나는 이번 달 기록을 끼니·음식군으로 걸러서 보고 싶다, 왜냐하면 원하는 항목만 빠르게 확인하고 싶기 때문이다. | 기능 4 (목록 필터/페이징) |
| US-6 | 사용자로서 나는 이번 달 총 섭취 칼로리와 끼니별·음식군별 칼로리를 보고 싶다, 왜냐하면 한 달 식습관 상태를 한눈에 파악하고 싶기 때문이다. | 기능 5 (월별 통계) |
| US-7 | 사용자로서 나는 내 데이터에만 접근하고 싶다, 왜냐하면 남이 내 식단 기록을 보거나 고치면 안 되기 때문이다. | 기능 6 (인가) |
| US-8 | (도전) 사용자로서 나는 월 목표 칼로리를 정하고 초과 시 경고를 받고 싶다, 왜냐하면 과다 섭취를 막고 싶기 때문이다. | 도전 (DailyGoal) |

## 3. 기능 요구사항 (MoSCoW)
| ID | 기능 | 설명 | 우선순위(Must/Should/Could) |
|----|------|------|:---:|
| F-01 | 회원가입/로그인 | JWT 발급, BCrypt | Must |
| F-02 | 카테고리 관리 | 시드(끼니: 아침/점심/저녁/간식, 음식군: 한식/양식/분식/디저트 등) + CRUD | Must |
| F-03 | 식단 기록 CRUD | 음식 등록·조회·수정·삭제 (음식명, 칼로리, 끼니, 음식군, 날짜) | Must |
| F-04 | 목록 필터·페이징 | yearMonth/mealType/foodGroupId + page/size | Must |
| F-05 | 월별 통계 | 총 섭취 칼로리, 끼니별·음식군별 칼로리 합계 | Must |
| F-06 | 인가 | 본인 데이터만 | Must |
| F-07 | 검증·예외 | 칼로리>0, 날짜 필수, 표준 에러 응답 | Must |
| F-08 | 프론트 4종 | 로그인/목록/등록/통계 | Must |
| F-09 | Swagger | API 문서화 | Should |
| F-10 | 배포 | Docker+GitHub Actions+EC2 | Must |
| F-11 | 월 목표 칼로리·초과 경고 | 월 단위 목표 칼로리(DailyGoal) 설정, 초과 시 경고 | Could (1순위) |
| F-12 | 통계 시각화 | 음식군별 파이차트, 월별 추이 라인차트 | Could (2순위) |
| F-13 | 검색·CSV 내보내기 | 키워드/기간/칼로리범위 검색, CSV(엑셀) 내보내기 | Could (2순위) |
| F-14 | Refresh Token 재발급 | JWT 재발급 흐름 | Could (1순위) |
| F-15 | 테스트·커버리지 | 단위/통합 테스트 + 커버리지 측정 | Could (1순위) |
| F-16 | Jenkins·K8s 배포 | Jenkins 파이프라인, K8s(Deployment/Service/Ingress), 롤링 업데이트 | Could (2순위) |
| F-17 | 비밀번호 찾기 | 이메일로 재설정 링크 발송 | Could (1순위) |
| F-18 | 이메일 인증 | 가입 시 이메일 인증 절차 | Could (2순위) |
| F-19 | 반복 식단 등록 | 정기 식단 자동 등록(예: 매일 아침 동일 메뉴) | Could (2순위) |
| F-20 | 연간 리포트 | 연도별 총 섭취 칼로리 집계 | Could (3순위) |
| F-21 | 소셜 로그인 | OAuth(Google 등) 연동 | Could (3순위) |
| F-22 | 음식 사진 첨부 | 파일 업로드·스토리지 연동 | Could (3순위) |
| F-23 | 식습관 예측 | 섭취 패턴 기반 예측 | Could (3순위) |
| F-24 | 알림 이메일/푸시 | 목표 초과 등 알림 발송 | Could (3순위) |
| F-25 | 모바일 앱 | Android/iOS 클라이언트 | Could (3순위) |
| F-26 | 오토스케일링/멀티리전 | 인프라 확장 | Could (3순위) |
| F-27 | 성능 부하 테스트 | JMeter 등으로 부하 검증 | Could (3순위) |

## 4. 비기능 요구사항
| 분류 | 요구사항 | 구체적 기준 |
|------|----------|-------------|
| 보안 | 본인 데이터만 | 모든 조회/수정/삭제 쿼리는 user_id = 로그인 사용자 조건 필수. 남의 기록 id로 접근 시 거부 |
| 보안 | 비밀번호 | 비밀번호는 평문 저장 금지, BCrypt 해시로 저장 |
| 보안 | 인증 방식 | 인증 필요 API는 Authorization: Bearer {accessToken} 헤더의 JWT로 검증 |
| 성능 | 목록 | 식단 기록 목록은 page/size 페이징으로 조회 (전체 로딩 금지) |
| 성능 | 통계 응답 | 월별 통계는 DB 집계 쿼리로 계산해 빠르게 응답 |
| 검증 | 입력 유효성 | calorie > 0, log_date 필수, mealType은 정해진 ENUM만 허용 |
| 품질 | 일관된 에러 응답 | 전역 예외 처리로 모든 에러를 {success:false, code, message, data:null} 형태로 통일 |
| 배포 | 실제 배포·접속 | AWS EC2에 배포하여 실제 접속 가능한 URL 제공 |
| 문서 | API 문서화 | Swagger UI로 API 명세 확인 가능 |
